## Prerequisites

Before you begin, install the following tools:

| Tool | Required version |
|------|------------------|
| Node.js | 22 or later (LTS recommended) |
| npm | 11.8.0 (pinned via `packageManager` field) |
| .NET SDK | 10.0.100 (`rollForward: latestFeature`) |
| SQL Server | 2019 or later (or a compatible Azure SQL instance) |

<Tip>
The backend pins the .NET SDK version in `backend/global.json`. Running `dotnet --version` inside the `backend/` directory will show `10.0.100` once the SDK is installed.
</Tip>

## Clone the repository

<Steps>
  <Step title="Clone and enter the project">
    <CodeGroup>
      ```bash git
      git clone https://github.com/your-org/cobranza-digital.git
      cd cobranza-digital
      ```
    </CodeGroup>
  </Step>

  <Step title="Configure the backend">
    The backend reads configuration from `appsettings.json` and environment-specific overrides. For local development, create `appsettings.Development.json` inside `backend/src/CobranzaDigital.Api/` with the following values:

    <CodeGroup>
      ```json appsettings.Development.json
      {
        "ConnectionStrings": {
          "DefaultConnection": "Server=localhost;Database=CobranzaDigitalDev;User Id=SA;Password=YourLocalPassword;TrustServerCertificate=true;"
        },
        "Jwt": {
          "Issuer": "CobranzaDigital",
          "Audience": "CobranzaDigital.Api",
          "SigningKey": "replace-with-a-secure-key-at-least-32-chars",
          "AccessTokenMinutes": 15,
          "RefreshTokenDays": 7
        },
        "Cors": {
          "AllowedOrigins": ["http://localhost:4200"]
        },
        "Swagger": {
          "Enabled": true
        },
        "Pos": {
          "RequireOpenShiftForSales": true
        }
      }
      ```
    </CodeGroup>

    <Warning>
      Never commit a real `SigningKey` value. The default `CHANGE_ME_TO_A_SECURE_32_CHAR_MIN_KEY` in `appsettings.json` is a placeholder and will cause JWT verification to fail if left unchanged.
    </Warning>

    To apply database migrations automatically on startup, set the environment variable `APPLY_MIGRATIONS_ON_STARTUP=1`. To seed initial development data (verticals, tenants, stores, and a default SuperAdmin user), also set `SEED_DEV_DATA=1`.
  </Step>

  <Step title="Run the backend">
    <CodeGroup>
      ```bash .NET CLI
      cd backend
      dotnet run --project src/CobranzaDigital.Api
      ```

      ```bash With dev seeding
      cd backend
      APPLY_MIGRATIONS_ON_STARTUP=1 SEED_DEV_DATA=1 dotnet run --project src/CobranzaDigital.Api
      ```
    </CodeGroup>

    The API starts on `https://localhost:5001` by default. Swagger UI is available at `https://localhost:5001/swagger` when `Swagger:Enabled` is `true`.

    Health endpoints:
    - `GET /health` — overall status
    - `GET /health/live` — liveness probe
    - `GET /health/ready` — readiness probe (includes database check)
  </Step>

  <Step title="Configure the frontend">
    Create or update `frontend/src/environments/environment.ts` to point at your running API:

    <CodeGroup>
      ```typescript environment.ts
      export const environment = {
        production: false,
        apiBaseUrl: '/api',
      };
      ```
    </CodeGroup>

    The Angular dev server proxies `/api` requests to the backend. Confirm your `proxy.conf.json` (or equivalent) targets `https://localhost:5001`.
  </Step>

  <Step title="Install frontend dependencies and start the dev server">
    <CodeGroup>
      ```bash npm
      cd frontend
      npm install
      npm start
      ```
    </CodeGroup>

    The Angular CLI serves the app at `http://localhost:4200`. The root path redirects to `/login`.
  </Step>

  <Step title="Open the application">
    Navigate to `http://localhost:4200`. You will see the login page.

    If you seeded development data (`SEED_DEV_DATA=1`), a default `SuperAdmin` account is created. Use those credentials to log in. After a successful login, `AuthService` stores the access token and refresh token in `localStorage` under `access_token` and `refresh_token`, then redirects you to `/app/dashboard`.
  </Step>
</Steps>

## Rate limiting

The API applies a fixed-window rate limiter per client IP address. The defaults from `appsettings.json` allow 100 requests per 60-second window. Requests that exceed the limit receive `429 Too Many Requests`. You can adjust these limits via `RateLimiting:PermitLimit`, `RateLimiting:WindowSeconds`, and `RateLimiting:QueueLimit`.
